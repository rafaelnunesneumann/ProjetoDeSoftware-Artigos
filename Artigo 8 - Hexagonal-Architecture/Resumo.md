Publicado originalmente por Alistair Cockburn em seu site pessoal por volta de 2005, o artigo "Hexagonal Architecture" descreve um padrão de design arquitetural também conhecido como Ports and Adapters. O objetivo central é simples de enunciar mas profundo de implicações: criar aplicações que possam ser igualmente acionadas por usuários, programas, scripts de teste automatizados ou processos em lote, e que possam ser desenvolvidas e testadas de forma completamente isolada dos dispositivos e bancos de dados que eventualmente irão usar em produção.

O problema que motivou o padrão é bem familiar para quem trabalha com sistemas há algum tempo: a lógica de negócio tende a vazar para a camada de interface com o usuário. Quando isso acontece, o sistema não consegue ser testado automaticamente com facilidade porque parte da lógica fica amarrada a detalhes visuais que mudam com frequência. Também fica impossível rodar o sistema em modo headless ou deixá-lo ser acionado por outro programa. A solução clássica de "criar uma nova camada" falha sistematicamente porque, sem um mecanismo que impeça a violação, poucos anos depois essa nova camada já está cheia de lógica de negócio e o problema se repete. O mesmo problema aparece no lado oposto da aplicação, quando a lógica de negócio se mistura com o banco de dados: se o banco sair do ar ou precisar ser trocado, os programadores ficam travados porque o código assumiu a existência de uma tecnologia específica.

A sacada do Cockburn foi perceber que esses dois problemas têm a mesma causa raiz: o entrelaçamento entre a lógica interna da aplicação e as entidades externas com as quais ela se comunica. A solução, então, não é tratar o lado do usuário e o lado do banco como problemas separados, mas reconhecer que existe uma assimetria fundamental entre o que está dentro e o que está fora da aplicação. A regra é clara: código pertencente ao lado interno não deve vazar para o lado externo.

A arquitetura funciona assim: a aplicação fica no centro de um hexágono. Cada face do hexágono representa uma porta, que define uma conversa com propósito específico. Para cada porta, pode haver múltiplos adaptadores: um adaptador converte a tecnologia externa (uma interface gráfica, um cliente HTTP, um framework de teste, um banco de dados SQL, um arquivo flat) para o protocolo da porta e vice-versa. A aplicação não precisa saber nada sobre a tecnologia do outro lado — ela conversa apenas com a porta.

O hexágono em si não foi escolhido por ter exatamente seis lados com algum significado especial. A forma foi adotada porque, ao contrário dos diagramas de camadas em uma dimensão, ela dá espaço visual para representar múltiplas portas sem impor uma hierarquia artificial de cima para baixo. A maioria dos sistemas reais tem de duas a quatro portas naturais: uma porta de controle do usuário ou sistema, uma porta de dados, eventualmente uma porta administrativa e uma porta de notificação.

Um detalhe importante do artigo é a distinção entre adaptadores primários e secundários, que Cockburn também chama de adaptadores de condução e adaptadores conduzidos. Um ator primário é aquele que inicia a conversa com a aplicação, como um usuário clicando na interface ou um script de teste mandando um comando. Um ator secundário é aquele que a aplicação aciona, como um banco de dados que responde a consultas ou um serviço externo que recebe notificações. Essa distinção importa na implementação: para portas primárias, é o ator externo que recebe uma referência para a aplicação e chama seus métodos. Para portas secundárias, é a aplicação que recebe e armazena uma referência para o ator externo, podendo chamá-lo quando necessário. Em ambos os casos, o que está sendo implementado é uma Dependência Configurável, só que em direções opostas.

O artigo traz um exemplo concreto bem ilustrativo: um sistema de cálculo de desconto que recebe um valor do usuário e consulta uma taxa num banco de dados. Cockburn mostra como implementar esse sistema em três estágios: primeiro com um framework de teste (FIT) acionando a aplicação e uma constante no lugar do banco; depois adicionando uma interface gráfica; por fim, introduzindo uma interface de repositório com uma implementação mock em memória e outra real com banco de dados. O resultado é uma aplicação que pode ser testada completamente isolada, demonstrada na interface, integrada ao banco de dados real e rodada em modo automatizado, tudo sem alterar o núcleo da lógica de negócio.

Cockburn também discute como os casos de uso se encaixam nessa visão: eles devem ser escritos na fronteira do hexágono interno, descrevendo funções e eventos suportados pela aplicação independentemente da tecnologia externa. Casos de uso escritos dessa forma são menores, mais legíveis, mais estáveis e mais baratos de manter do que aqueles que assumem conhecimento sobre as tecnologias de cada porta.

No campo dos padrões relacionados, o artigo conecta a arquitetura hexagonal com o padrão Adapter do GoF (que é exatamente o que os adaptadores implementam), com MVC (que cobre apenas as portas primárias, não as secundárias), com Mock Objects (que são a implementação natural de adaptadores secundários nos testes), com o padrão Loopback (substituto interno para dispositivos externos) e com o Princípio de Inversão de Dependência de Bob Martin, que fornece a base teórica para os adaptadores secundários.

Entre os usos conhecidos, Cockburn cita aplicações MacIntosh e Windows que expunham APIs para scripts automatizados, a tendência de Web 2.0 de publicar APIs abertas (o que permite mashups como sobrepor dados de crimes num mapa do Google), e um caso real de um sistema de alertas climáticos que precisava suportar múltiplas tecnologias de entrada e saída. Nesse último caso, a virada foi repensar as interfaces por propósito em vez de por tecnologia, o que permitiu adicionar novas formas de entrada e saída apenas criando novos adaptadores, sem modificar o núcleo.

O artigo é uma leitura fundamental para quem quer entender a genealogia das arquiteturas modernas. A Arquitetura Hexagonal influenciou diretamente a Onion Architecture de Jeffrey Palermo (2008) e a Clean Architecture de Robert C. Martin (2012), que aprofundam e refinam os mesmos princípios com camadas concêntricas e regras estritas de dependência. Também há quem argumente que ela está na raiz da arquitetura de microsserviços, já que cada serviço pode ser pensado como um hexágono com portas bem definidas. A ideia de isolar completamente a lógica de negócio das preocupações de infraestrutura, que Cockburn formaliza aqui de forma elegante, é hoje um princípio amplamente aceito no design de software.
